# SandboxFusion - Repository Summary

## Overview

SandboxFusion is a **secure, multi-language code execution sandbox** originally built by Bytedance for running and evaluating code generated by LLMs. It serves as the execution backend for RL training loops (reward computation via test-case pass/fail) and online judge workflows. The project is licensed under Apache 2.0.

The system has two primary capabilities:
1. **Code Runner** - Execute arbitrary code snippets in 28 CPU languages and return stdout/stderr/return-code.
2. **Evaluation** - Evaluate LLM-generated code completions against caller-supplied stdin/stdout test cases. No database, no pre-loaded datasets — test cases are passed inline with each request.

## Architecture

### Layered Design

```
                          ┌─────────────────────────────┐
 Client SDK / HTTP  ────> │  FastAPI Server (server.py)  │
                          ├──────────────┬──────────────┤
                          │ sandbox_api  │  submit_api   │
                          │ POST         │  POST         │
                          │ /run_code    │  /submit      │
                          ├──────────────┴──────────────┤
                          │     Runners (28 CPU langs)   │
                          │  CODE_RUNNERS[language]()    │
                          │  → run_commands()            │
                          ├─────────────────────────────┤
                          │     Isolation Layer          │
                          │  lite: overlayfs + cgroups   │
                          │        + netns + chroot      │
                          │  full: Docker containers     │
                          └─────────────────────────────┘
```

### Key Data Flow

**Code Execution (`POST /run_code`):**
1. `RunCodeRequest` arrives (code, language, timeouts, optional files)
2. Dispatched to the appropriate runner function via `CODE_RUNNERS[language]`
3. Runner writes code to a temp directory, constructs compile + run commands
4. `run_commands()` executes via `asyncio.create_subprocess_shell` with timeout enforcement
5. In `lite` mode: wraps in overlayfs + cgroup + chroot + network namespace
6. In `full` mode: wraps in `docker run --rm` with `--memory`, `--cpus`, `--network none`, `--pids-limit`
7. Returns `RunCodeResponse` with compile/run results, stdout, stderr, return code

**Evaluation (`POST /submit`):**
1. `SubmitRequest` arrives (id, completion, config with language, test_cases as inline `List[GeneralStdioTest]`)
2. Completion is extracted using `default_extract_helper()` (fenced code blocks, heuristics)
3. Each test case is run in parallel via `check_stdio_test_cases_parallel()` with semaphore concurrency control
4. Pass/fail determined by comparing stdout against expected output
5. `EvalResult` returned with per-test-case results and `accepted` (all passed)

### Isolation Modes

| Mode | Mechanism | Overhead | Use Case |
|------|-----------|----------|----------|
| `lite` | overlayfs (copy-on-write FS) + cgroups v1 (memory/CPU limits) + network namespaces + PID namespace (`unshare`) + `chroot` | ~100ms | Production on Linux, CI, untrusted code |
| `full` | Docker containers with `--memory`, `--cpus 1`, `--network none`, `--pids-limit 256` | ~500ms+ | Any platform with Docker, strongest isolation |

Both modes require Linux. `lite` requires root (sudo) for overlayfs/cgroups. `full` requires a running Docker daemon and the configured `docker_image`.

### Configuration

Controlled by YAML files selected via `SANDBOX_CONFIG` env var (default: `local`):
- **`local.yaml`**: Lite isolation, max 34 concurrent runners, max 3 eval runner concurrency
- **`ci.yaml`**: Lite isolation, max concurrency 0 (unlimited, controlled by pytest-xdist workers)

Key config sections:
- `sandbox.isolation`: `lite` | `full`
- `sandbox.max_concurrency`: Limits simultaneous code executions
- `sandbox.docker_image`: Docker image for `full` mode (default: `sandbox:base`)
- `eval.max_runner_concurrency`: Limits parallel test case runners (0 = unlimited)
- `common.logging_color`: Colored structlog output

### Request / Response Types

**`SubmitRequest`** (dataset-agnostic, no database):
```python
class SubmitRequest(BaseModel):
    id: int | str
    completion: str           # Raw LLM output (code is extracted automatically)
    config: TestConfig        # language, timeouts, custom_extract_logic
    test_cases: List[GeneralStdioTest]  # Inline stdin/stdout pairs
```

**`GeneralStdioTest`**:
```python
class GeneralStdioTest(BaseModel):
    input: Dict[str, str]     # {"stdin": "1 2\n"}
    output: Dict[str, str]    # {"stdout": "3\n"}
```

**`EvalResult`**:
```python
class EvalResult(BaseModel):
    id: int | str
    accepted: bool            # True if ALL test cases passed
    extracted_code: str       # Code extracted from completion
    tests: List[EvalTestCase] # Per-test-case results
```

## Common Modes of Operation & Invocation

### 1. Running the Server (Development)

```bash
conda create -n sandbox -y python=3.13
conda activate sandbox
poetry install
mkdir -p docs/build

make run          # hot-reload mode on 0.0.0.0:8080
make run-online   # production-like (no reload)
```

Server listens on `http://0.0.0.0:8080` with endpoints:
- `GET  /` - Redirect to documentation site
- `GET  /v1/ping` - Health check (returns `"pong"`)
- `POST /run_code` - Execute code in any supported CPU language
- `POST /submit` - Submit a completion for evaluation against inline test cases

### 2. Running via Docker (Production)

```bash
make build-base-image                # builds sandbox:base
make build-server-image              # builds sandbox:server
docker run -d --rm --privileged -p 8080:8080 sandbox:server
```

### 3. Using the Python Client SDK

```python
from sandbox_fusion import client
from sandbox_fusion.models import RunCodeRequest, SubmitRequest, TestConfig

client.set_endpoint("http://localhost:8080")

# Execute code
result = client.run_code(RunCodeRequest(
    code='print("hello world")',
    language='python',
    run_timeout=10
))

# Evaluate a completion against inline test cases
eval_result = client.submit(SubmitRequest(
    id='problem-1',
    completion='```python\na, b = map(int, input().split())\nprint(a + b)\n```',
    config=TestConfig(language='python'),
    test_cases=[
        {"input": {"stdin": "1 2\n"}, "output": {"stdout": "3\n"}},
        {"input": {"stdin": "10 20\n"}, "output": {"stdout": "30\n"}},
    ]
))
print(eval_result.accepted)  # True or False
```

### 4. Direct HTTP API Usage

```bash
# Health check
curl http://localhost:8080/v1/ping

# Execute code
curl -X POST http://localhost:8080/run_code \
  -H "Content-Type: application/json" \
  -d '{"code": "print(sum(range(10)))", "language": "python", "run_timeout": 10}'

# Submit completion for evaluation
curl -X POST http://localhost:8080/submit \
  -H "Content-Type: application/json" \
  -d '{
    "id": "problem-1",
    "completion": "```python\na, b = map(int, input().split())\nprint(a + b)\n```",
    "config": {"language": "python"},
    "test_cases": [
      {"input": {"stdin": "1 2\n"}, "output": {"stdout": "3\n"}},
      {"input": {"stdin": "10 20\n"}, "output": {"stdout": "30\n"}}
    ]
  }'
```

### 5. RL Training Loop Pattern

1. **Prepare problems** with stdin/stdout test cases (from any source)
2. **Generate completions** by feeding prompts to the LLM being trained
3. **Submit completions** via `POST /submit` with inline test cases to get pass/fail reward signal
4. The `accepted` field in `EvalResult` serves as the binary reward signal
5. Per-test-case results in `tests` provide fine-grained feedback

### 6. Running Tests

```bash
make test                        # All tests with 4 parallel workers
make test-case CASE=test_python  # Single test with stdout
make test-minor                  # Minor language tests only
```

### 7. Configuration Override

```bash
SANDBOX_CONFIG=ci make run-online    # Use ci.yaml
SANDBOX_CONFIG=local make run        # Use local.yaml (default)
```
