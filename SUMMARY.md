# SandboxFusion - Repository Summary

## Overview

SandboxFusion is a **secure, multi-language code execution sandbox** built by Bytedance for running and evaluating code generated by LLMs. It serves as the execution backend for RL training loops (reward computation via test-case pass/fail) and online judge workflows. The project is licensed under Apache 2.0.

The system has two primary capabilities:
1. **Code Runner** - Execute arbitrary code snippets in 28 CPU languages and return stdout/stderr/return-code.
2. **Online Judge** - Evaluate LLM-generated code completions against stdin/stdout test cases using the CommonOJDataset format (e.g., CodeContests).

## Architecture

### Layered Design

```
                          ┌─────────────────────────────┐
 Client SDK / HTTP  ────> │  FastAPI Server (server.py)  │
                          ├──────────────┬──────────────┤
                          │ sandbox_api  │  oj_api      │
                          │ /run_code    │  /submit      │
                          │              │  /get_prompts │
                          ├──────────────┴──────────────┤
                          │     Registry + Database      │
                          │  (dynamic class loading,     │
                          │   MySQL / in-memory SQLite)  │
                          ├─────────────────────────────┤
                          │     Runners (28 CPU langs)   │
                          │  run_command_bare() ->        │
                          │  asyncio subprocess          │
                          ├─────────────────────────────┤
                          │     Isolation Layer          │
                          │  overlayfs + cgroups + netns │
                          │  (or none for dev)           │
                          └─────────────────────────────┘
```

### Key Data Flow

**Code Execution (`/run_code`):**
1. `RunCodeRequest` arrives (code, language, timeouts, optional files)
2. Dispatched to the appropriate runner function via `CODE_RUNNERS[language]`
3. Runner writes code to a temp directory, constructs compile + run commands
4. `run_commands()` executes via `asyncio.create_subprocess_shell` with timeout enforcement
5. In `lite` isolation mode: wraps execution in overlayfs + cgroup + chroot + network namespace
6. Returns `RunCodeResponse` with compile/run results, stdout, stderr, return code

**Evaluation (`/submit`):**
1. `SubmitRequest` arrives (dataset, problem id, LLM completion, config with language)
2. Registry resolves `CommonOJDataset` class
3. Dataset fetches problem row from DB (test cases as stdin/stdout pairs)
4. Completion is extracted using `default_extract_helper()` (fenced code blocks, heuristics)
5. Each test case is run in parallel via `check_stdio_test_cases_parallel()`
6. Pass/fail determined by comparing stdout against expected output
7. `EvalResult` returned with per-test-case results

### Isolation Modes

| Mode | Config | Mechanism | Overhead | Use Case |
|------|--------|-----------|----------|----------|
| `none` | `local.yaml` | No isolation; best-effort process cleanup, optional `setuid`, `resource.setrlimit` for memory | ~0ms | Local development, trusted code |
| `lite` | `ci.yaml` | overlayfs (copy-on-write FS) + cgroups (memory/CPU limits) + network namespaces + `chroot` + PID namespace (`unshare`) | ~100ms | Production, CI, untrusted code |

### Configuration

Controlled by YAML files selected via `SANDBOX_CONFIG` env var (default: `local`):
- **`local.yaml`**: No isolation, no DB backend, loads sample JSONL files into in-memory SQLite, max 34 concurrent runners
- **`ci.yaml`**: Lite isolation, max concurrency controlled by pytest-xdist workers

Key config sections:
- `sandbox.isolation`: `none` | `lite`
- `sandbox.max_concurrency`: Limits simultaneous code executions
- `dataset.database.backend`: `mysql` (production) | `none` (local/test)
- `dataset.database.cache`: In-memory SQLite populated from local JSONL or MySQL tables
- `dataset.registry`: CommonOJDataset with module path and table mappings

### Dataset System

The sole dataset type is `CommonOJDataset`, which extends `CodingDataset` (abstract base class):
- `get_prompts(request)` - Paginated prompt retrieval from DB
- `get_prompt_by_id(request)` - Single prompt lookup
- `evaluate_single(request)` - Extract code, run against all stdin/stdout test cases in parallel, return pass/fail

Problems are stored as rows with `id`, `labels`, `content` (prompt text), and `test` (JSON array of `{input: {stdin: ...}, output: {stdout: ...}}` pairs).

## Common Modes of Operation & Invocation

### 1. Running the Server (Development)

```bash
conda create -n sandbox -y python=3.12
conda activate sandbox
poetry install
mkdir -p docs/build

make run          # hot-reload mode
make run-online   # production-like (no reload)
```

Server listens on `http://0.0.0.0:8080` with endpoints:
- `GET  /v1/ping` - Health check
- `POST /run_code` - Execute code in any supported CPU language
- `GET  /list_datasets` - List registered datasets
- `POST /list_ids` - List problem IDs in a dataset
- `POST /get_prompts` - Get prompts (paginated)
- `POST /get_prompt_by_id` - Get a single prompt
- `POST /submit` - Submit a completion for evaluation
- `POST /get_metrics` - Compute aggregate metrics

### 2. Running via Docker (Production)

```bash
docker build -f ./scripts/Dockerfile.base -t code_sandbox:base .
sed -i '1s/.*/FROM code_sandbox:base/' ./scripts/Dockerfile.server
docker build -f ./scripts/Dockerfile.server -t code_sandbox:server .
docker run -d --rm --privileged -p 8080:8080 code_sandbox:server make run-online
```

### 3. Using the Python Client SDK

```python
from sandbox_fusion import client

client.set_endpoint("http://localhost:8080")

# Execute code
from sandbox_fusion.models import RunCodeRequest
result = client.run_code(RunCodeRequest(
    code='print("hello world")',
    language='python',
    run_timeout=10
))

# Evaluate a completion against CodeContests
from sandbox_fusion.models import SubmitRequest, TestConfig
eval_result = client.submit(SubmitRequest(
    dataset='code_contests',
    id=17,
    completion='```python\nimport sys\nprint(sys.stdin.read())\n```',
    config=TestConfig(language='python')
))
print(eval_result.accepted)  # True or False
```

### 4. Direct HTTP API Usage (RL Training Loop)

```bash
# Execute code
curl -X POST http://localhost:8080/run_code \
  -H "Content-Type: application/json" \
  -d '{"code": "print(sum(range(10)))", "language": "python", "run_timeout": 10}'

# Submit completion for evaluation
curl -X POST http://localhost:8080/submit \
  -H "Content-Type: application/json" \
  -d '{"dataset": "code_contests", "id": 17, "completion": "...", "config": {"language": "python"}}'

# Get prompts
curl -X POST http://localhost:8080/get_prompts \
  -H "Content-Type: application/json" \
  -d '{"dataset": "code_contests", "config": {}, "offset": 0, "limit": 10}'
```

### 5. RL Training Loop Pattern

1. **Fetch prompts** via `POST /get_prompts` for the target dataset
2. **Generate completions** by feeding prompts to the LLM being trained
3. **Submit completions** via `POST /submit` to get pass/fail reward signal
4. **Compute metrics** via `POST /get_metrics` to track aggregate progress
5. The `accepted` field in `EvalResult` serves as the binary reward signal

For custom/inline data (no pre-loaded dataset), use `config.provided_data` to pass problem data directly in the request, bypassing the database entirely.

### 6. Running Tests

```bash
make test                        # All tests with 4 parallel workers
make test-case CASE=test_python  # Single test with stdout
make test-minor                  # Minor language tests only
```

### 7. Configuration Override

```bash
SANDBOX_CONFIG=ci make run-online    # Use ci.yaml (lite isolation)
SANDBOX_CONFIG=local make run        # Use local.yaml (no isolation, default)
```
